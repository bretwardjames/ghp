# Deploying ghp-mcp-hosted to Railway

Step-by-step for shipping a production instance. Assumes you already have a GitHub account and a Railway account (https://railway.app).

## 1. Create the GitHub OAuth App

1. Go to https://github.com/settings/developers.
2. **New OAuth App.**
3. Fill in:
   - **Application name:** `GHP MCP (your-subdomain)` — something recognizable to end users on the consent screen.
   - **Homepage URL:** the URL end users would click to learn more (e.g., your docs page or `https://<your-railway-subdomain>.up.railway.app`).
   - **Authorization callback URL:** `https://<your-railway-subdomain>.up.railway.app/oauth/callback`
     You will not know the exact subdomain until after Railway deploys; come back and update this value.
4. **Register application.** Railway's autogenerated subdomain takes the form `<project-name>-production.up.railway.app`. If you intend to use a custom domain, set the callback to that instead and add the Railway subdomain as a second authorized callback during testing.
5. Click **Generate a new client secret**. Copy both the **Client ID** and the **Client secret** — you will paste them into Railway in step 3.

## 2. Deploy to Railway

1. **New Project → Deploy from GitHub repo.** Select `bretwardjames/ghp`.
2. In the service settings, under **Build**, Railway should auto-detect `packages/mcp-hosted/railway.json` once it exists on the default branch. If not, set **Config path** to `packages/mcp-hosted/railway.json` manually.
3. Railway will build the Dockerfile at `packages/mcp-hosted/Dockerfile`. Build context is the repo root.

## 3. Configure environment

In the Railway service's **Variables** tab, add:

| Variable | Value |
|----------|-------|
| `GHP_MCP_MODE` | `hosted` |
| `GHP_REPO` | e.g. `bretwardjames/ghp` — single repo per instance |
| `GHP_GITHUB_OAUTH_CLIENT_ID` | from step 1 |
| `GHP_GITHUB_OAUTH_CLIENT_SECRET` | from step 1 |
| `GHP_ALLOWED_REDIRECT_URIS` | the exact `redirect_uri` your MCP client uses, e.g. `https://app.runtight.io/api/mcp-servers/oauth/callback`. Comma-separated for multiple. |
| `GHP_HOSTED_BASE_URL` | `https://<your-subdomain>.up.railway.app` — must be https in production |
| `NODE_ENV` | `production` |

Optional:

| Variable | Default | Purpose |
|----------|---------|---------|
| `PORT` | Railway injects | Listen port |
| `GHP_ALLOWED_ORIGINS` | `*` | CORS allowlist. Tighten for production if runtight is the only consumer. |
| `GHP_OAUTH_STATE_TTL_SECONDS` | `600` | Ephemeral authorize / auth-code TTL |

## 4. First deploy

1. Click **Deploy**. Railway builds the image (~2 min cold, ~30s warm) and starts it.
2. Confirm `/healthz` returns `ok`:
   ```bash
   curl https://<your-subdomain>.up.railway.app/healthz
   # ok
   ```
3. Confirm OAuth metadata is live:
   ```bash
   curl https://<your-subdomain>.up.railway.app/.well-known/oauth-authorization-server | jq
   ```

## 5. Go back to GitHub and fix the callback URL

If you deployed with a placeholder callback in step 1.4, return to https://github.com/settings/developers and set the **Authorization callback URL** to the real Railway URL + `/oauth/callback`. Save.

## 6. Register with your MCP client

For runtight, follow the steps in ticket #281. Short version: `POST /api/mcp-servers` with:

```json
{
  "name": "GHP",
  "slug": "ghp",
  "connectionUrl": "https://<your-subdomain>.up.railway.app/mcp",
  "transportType": "streamable-http",
  "authType": "per_user",
  "scope": "account"
}
```

Then click Connect in runtight's UI → consent on github.com → see "Connected as @username" in runtight.

## Local testing alternative (no Railway)

For testing OAuth flows without deploying, use Tailscale Funnel:

```bash
tailscale funnel --bg --https=8443 http://localhost:8731

GHP_MCP_MODE=hosted \
GHP_REPO=bretwardjames/ghp \
GHP_GITHUB_OAUTH_CLIENT_ID=<from-github> \
GHP_GITHUB_OAUTH_CLIENT_SECRET=<from-github> \
GHP_ALLOWED_REDIRECT_URIS=<your-mcp-client-redirect> \
GHP_HOSTED_BASE_URL=https://<machine>.tail-xxx.ts.net:8443 \
  pnpm --filter @bretwardjames/ghp-mcp-hosted start
```

GitHub OAuth App callback for this path: `https://<machine>.tail-xxx.ts.net:8443/oauth/callback`.

## Troubleshooting

**Build fails with `ERR_PNPM_NO_LOCKFILE`.** Ensure build context is the repo root. `railway.json`'s `dockerfilePath` already points at `packages/mcp-hosted/Dockerfile`; Railway sets context to the repo root by default. If running locally, run from the root:
```bash
docker build -f packages/mcp-hosted/Dockerfile -t ghp-mcp-hosted .
```

**Image layout:** the runtime container uses `pnpm deploy` output rather than the full monorepo. Inside the container, the working directory is `/app` and the entrypoint is `node dist/bin.js` (not `packages/mcp-hosted/dist/bin.js`). Workspace deps (`@bretwardjames/ghp-core`, `@bretwardjames/ghp-mcp`) are materialised as real `node_modules/` directories instead of symlinks.

**`/oauth/authorize` returns 400 `invalid_redirect_uri`.** The MCP client's `redirect_uri` must be an exact match (including trailing slash, host case, default ports) of an entry in `GHP_ALLOWED_REDIRECT_URIS`. Copy/paste the exact string from the client's configuration.

**GitHub consent screen says "The redirect_uri MUST match the registered callback URL".** Update the callback URL on your GitHub OAuth App to exactly `<GHP_HOSTED_BASE_URL>/oauth/callback`. GitHub is strict about scheme and host.

**HEALTHCHECK never succeeds.** Confirm `PORT` matches what the container listens on. Railway usually sets `PORT`; if absent the server defaults to 8731. If Railway's health check is pointed elsewhere, verify `healthcheckPath` in `railway.json` is `/healthz`.
